# 2장. 개략적인 규모 추정

> 목적: 시스템 설계 면접에서 **트래픽, 저장소, 지연시간, 가용성**을 빠르게 근사 계산해 설계 방향을 잡는다.  
> 정확한 숫자가 아니라 **어떤 가정을 세우고 어떤 순서로 추정하는지**가 중요하다.

## 2-1. 개략적인 규모 추정이란?

면접에서는 "초당 몇 요청을 처리해야 하나, 저장소·캐시·서버는 얼마나 필요한가" 같은 질문을 받는다.   
핵심은 정확한 숫자가 아니라 **합리적인 가정을 세우고 계산 과정을 명확히 보여주는 것**이다. 

예: 트위터 같은 서비스를 설계
- 하루에 트윗이 몇 개 생기는가?
- 초당 쓰기 요청은 몇 개인가?
- 미디어 파일은 하루에 얼마나 쌓이는가?
- 5년 보관하면 저장소가 얼마나 필요한가?

즉, 개략적인 규모 추정은 **설계 방향을 잡기 위한 숫자 감각**이다.

## 2-2. 2의 제곱수 감각

핵심은 `2¹⁰ ≈ 10³`이고, 따라서 KB → MB → GB → TB → PB는 각각 **약 1000배씩** 커진다. 정확히는 1024배지만 면접에서는 1000으로 근사한다.

| 2의 제곱 |  대략적인 값 | 단위  |
|------:|--------:|-----|
|   2¹⁰ |    약 1천 | 1KB |
|   2²⁰ |  약 100만 | 1MB |
|   2³⁰ |   약 10억 | 1GB |
|   2⁴⁰ |    약 1조 | 1TB |
|   2⁵⁰ | 약 1000조 | 1PB |

## 2-3. 응답지연 값

숫자 자체보다 **상대적인 크기 차이**가 중요하다.

```text
캐시 < 메모리 < 네트워크 < 디스크(SSD/HDD) < 다른 지역 데이터센터
```

| 연산                | 대략적인 시간 |
|-------------------|--------:|
| L1 캐시 참조          |   0.5ns |
| 주 메모리 참조          |   100ns |
| 1KB 압축            |    10μs |
| 같은 데이터센터 내 왕복     |   500μs |
| 디스크 탐색(seek)      |    10ms |
| 디스크에서 1MB 순차 읽기   |    30ms |
| 캘리포니아 ↔ 네덜란드 왕복   |   150ms |

> 단위: 1ms = 1,000μs = 1,000,000ns

**기억할 결론 네 가지**

1. 메모리는 빠르고 디스크는 느리다.
2. 디스크 탐색(seek)은 피해야 한다 — 순차 읽기/캐싱/인덱싱으로 줄인다.
3. 네트워크 전송 전 압축하면 비용을 줄일 수 있다.
4. 데이터센터 간 통신은 내부 통신보다 훨씬 느리다.

## 2-4. 가용성에 관계된 수치들

### 고가용성(High Availability)
- 시스템이 중단 없이 운영되는 능력을 말한다
- 퍼센트로 표현한다

```text
100% 가용성 = 한 번도 중단되지 않음
```

관습적으로 숫자 9를 사용한다. **9가 하나 늘 때마다**(가용성이 높아질 수록) **장애 허용 시간은 급격히 줄고, 비용과 설계 난이도는 크게 오른다.**


|      가용성 | 하루 장애 시간 | 주간 장애 시간 | 월간 장애 시간 | 연간 장애 시간 |
|---------:|---------:|---------:|---------:|---------:|
|      99% |   14.40분 |   1.68시간 |   7.31시간 |    3.65일 |
|    99.9% |    1.44분 |   10.08분 |   43.83분 |   8.77시간 |
|   99.99% |    8.64초 |    1.01분 |    4.38분 |   52.60분 |
|  99.999% |    864ms |    6.05초 |   26.30초 |    5.26분 |
| 99.9999% |   86.4ms |  604.8ms |    2.63초 |   31.56초 |


### SLA란?

**SLA(서비스 수준 협약)** 는 제공자가 보장하는 가용성 계약이다(예: "월 99.99% 보장"). 요구 SLA에 따라 아키텍처가 달라진다. 99.999%를 맞추려면 단일 장애점 제거, 복제, 장애 감지, 자동 복구, 멀티 리전 같은 요소가 필요하다.

## 2-5. 예제: 트위터 QPS와 저장소 추정

실제 수치가 아닌 연습용 가정이다.

### 가정

```text
MAU = 3억 명
매일 사용하는 비율 = 50%
사용자당 하루 트윗 = 2개
미디어 포함 비율 = 10%, 미디어 평균 크기 = 1MB
보관 기간 = 5년
```

### 추정 계산
#### DAU 계산

MAU가 3억 명이고, 그중 50%가 매일 사용한다고 가정한다.

```text
DAU = 3억 × 50%
    = 1.5억 명
```

즉, 하루 활성 사용자는 약 **1.5억 명**이다.

#### 하루 트윗 수 계산

사용자 한 명이 하루 평균 2개의 트윗을 올린다고 가정한다.

```text
하루 트윗 수 = 1.5억 × 2
           = 3억 개
```

즉, 하루에 약 **3억 개의 트윗**이 생성된다.

#### 평균 QPS 계산

QPS는 **Query Per Second**, 즉 초당 요청 수다.

하루는 24시간이고, 초로 바꾸면 다음과 같다.

```text
24시간 × 3600초 = 86,400초
```

따라서 평균 쓰기 QPS는 다음과 같이 계산한다.

```text
QPS = 하루 트윗 수 / 하루 초 수
    = 3억 / 86,400
    ≈ 3,500
```

즉, 평균적으로 초당 약 **3,500개의 트윗 작성 요청**이 들어온다.

면접에서는 86,400초를 대략 100,000초로 보고 빠르게 근사해도 된다.

```text
3억 / 100,000 ≈ 3,000
```

정확도보다는 자릿수 감각이 중요하다.

#### 최대 QPS 계산

피크 시간대에는 평균보다 요청이 많다고 가정한다.

책에서는 단순히 평균 QPS의 2배로 잡는다.

```text
최대 QPS = 평균 QPS × 2
        ≈ 3,500 × 2
        ≈ 7,000
```

즉, 피크 QPS는 약 **7,000**으로 추정한다.

#### 미디어 저장소 요구량 계산

트윗 중 10%가 미디어를 포함하고, 미디어 하나가 평균 1MB라고 가정한다.

하루 트윗 수는 3억 개였으므로 다음과 같이 계산한다.

```text
하루 미디어 트윗 수 = 3억 × 10%
                  = 3천만 개
```

미디어 하나가 1MB라면 다음과 같다.

```text
하루 미디어 저장소 = 3천만 × 1MB
                 = 30,000,000MB
                 ≈ 30TB
```

즉, 하루에 약 **30TB**의 미디어 저장소가 필요하다.

#### 5년 보관 시 저장소 계산

데이터를 5년 보관한다고 가정한다.

```text
5년 저장소 = 30TB × 365 × 5
          = 54,750TB
          ≈ 55PB
```

즉, 미디어만 저장해도 약 **55PB**가 필요하다.

여기서 중요한 통찰은 다음과 같다.

> 텍스트 트윗보다 미디어 파일이 저장소 요구량을 압도한다.

`tweet_id` 64바이트, 텍스트 140바이트 정도는 미디어 1MB에 비하면 매우 작다. 대규모 서비스에서는 어떤 데이터가 용량을 지배하는지 빨리 찾는 것이 중요하다.


## 2-6. 면접 팁

### 정확한 계산보다 과정이 중요하다

면접관은 정답 숫자 하나를 보려는 것이 아니라, **문제를 풀어가는 방식**을 본다.

좋은 답변 예시는 다음과 같다.

```text
먼저 사용자 수를 가정하고,
그중 일간 활성 사용자를 계산한 뒤,
사용자당 요청 수를 곱해서 일일 요청량을 구하겠습니다.
그다음 초 단위로 나누어 평균 QPS를 구하고,
피크는 평균의 2배로 잡겠습니다.
```

이렇게 말하면 계산이 조금 틀려도 사고 과정이 명확하다.

### 근사치를 활용하여 계산하기

복잡한 계산에 시간을 쓰면 안 된다.

예를 들어, 다음 계산을 정확히 하려고 할 필요는 없다.

```text
99987 / 9.1
```

대신 이렇게 근사한다.

```text
100,000 / 10 = 10,000
```

시스템 설계 면접에서는 소수점 몇 자리보다 **자릿수 감각**이 중요하다.

### 가정을 적어두라

나중에 계산을 검토할 수 있어야 한다.

### 단위를 반드시 붙여라

숫자만 쓰면 의미가 없다.

```text
5 -> 5KB, 5MB, 5GB, 5TB
```

시스템 설계에서는 단위 실수가 치명적이다.

### 자주 나오는 추정 항목

시스템 설계 면접에서 자주 나오는 추정 항목은 다음과 같다.

| 항목       | 의미                     |
|----------|------------------------|
| QPS      | 평균 초당 요청 수             |
| Peak QPS | 피크 시간대 초당 요청 수         |
| 저장소 요구량  | 하루/월/년 단위 데이터 증가량      |
| 캐시 요구량   | 자주 읽히는 데이터를 얼마나 캐싱할지   |
| 서버 수     | 필요한 처리량을 감당하기 위한 서버 대수 |
| 대역폭      | 네트워크로 오가는 데이터 양        |
| 가용성      | 장애 허용 시간과 SLA          |
