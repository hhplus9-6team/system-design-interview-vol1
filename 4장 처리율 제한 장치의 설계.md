# 4장. 처리율 제한 장치의 설계

## 1. 처리율 제한 장치란

처리율 제한 장치는 사용자의 요청 수를 일정 기준 이하로 제한하는 장치이다.

예를 들어 사용자가 1초에 10번까지만 API를 호출할 수 있도록 제한하거나, 특정 IP가 1분에 100번 이상 요청하지 못하도록 막는 방식이다.

특히 외부에 공개된 API나 트래픽이 많은 서비스에서는 필수적인 보호 장치이다.

---
## 2. 처리율 제한 장치가 필요한 이유

처리율 제한 장치가 필요한 이유는 크게 세 가지이다.

1. 서비스 안정성을 보호하기 위해서이다. 특정 사용자나 봇이 과도한 요청을 보내면 서버 자원이 고갈될 수 있다. 처리율 제한 장치는 이런 과도한 요청을 차단하여 전체 시스템이 안정적으로 동작하도록 돕는다.
2. 비용을 절감하기 위해서이다. 서버 자원, 데이터베이스 연결, 외부 API 호출 등은 모두 비용과 연결된다. 요청 수를 제한하면 불필요한 자원 사용을 줄일 수 있다. 
3. 악의적인 공격을 완화하기 위해서이다. 무차별 대입 공격, 크롤링, DDoS성 요청 등을 완전히 막을 수는 없지만, 요청량을 제한함으로써 피해를 줄일 수 있다.

---
## 3. 처리율 제한 장치의 위치

### 3.1. 클라이언트 측 
클라이언트에서 요청 빈도를 제한할 수 있지만, 클라이언트는 사용자가 조작할 수 있으므로 신뢰하기 어렵다. 따라서 보안 목적의 처리율 제한에는 적합하지 않다.

### 3.2. 서버
#### 1) 서버 애플리케이션 내부
Spring 애플리케이션에서 필터, 인터셉터, AOP 등을 이용해 요청 수를 제한할 수 있다. 구현이 비교적 쉽고 비즈니스 조건에 맞춘 제한이 가능하다는 장점이 있다. 
그러나 서버가 여러 대일 경우 각 서버가 요청 수를 따로 계산하면 전체 제한이 정확하지 않을 수 있다.

#### 2) 미들웨어 또는 게이트웨이 계층 
API Gateway, Nginx, 로드밸런서, Redis 기반 공통 제한 장치 등을 활용할 수 있다. 
여러 서버 앞단에서 요청을 제어할 수 있으므로 실무에서 자주 사용되는 방식이다.

---
## 4. 처리율 제한 기준

대표적인 기준은 사용자 ID, IP 주소, API Key, 엔드포인트, 서비스 플랜 등이다.

로그인한 사용자의 요청을 제한할 때는 사용자 ID를 기준으로 삼을 수 있다. 로그인하지 않은 요청은 IP 주소를 기준으로 삼는 경우가 많다. 외부 개발자에게 제공하는 API라면 API Key를 기준으로 제한할 수 있다.

또한 모든 API에 동일한 제한을 적용하는 것이 아니라, API별로 다른 제한을 둘 수도 있다. 예를 들어 단순 조회 API는 분당 100회까지 허용하고, 결제 API는 분당 10회까지만 허용하는 식이다.

## 5. 처리율 제한 알고리즘

### 5.1 토큰 버킷 알고리즘

토큰 버킷 알고리즘은 일정한 속도로 버킷에 토큰을 추가하고, 요청이 들어올 때마다 토큰을 하나씩 소비하는 방식이다.

버킷에 토큰이 있으면 요청을 허용하고, 토큰이 없으면 요청을 거부한다.

이 방식은 일시적인 트래픽 증가를 어느 정도 허용할 수 있다는 장점이 있다. 예를 들어 버킷 크기가 10이고 초당 1개의 토큰이 추가된다면, 사용자는 순간적으로 최대 10개의 요청을 보낼 수 있다. 이후에는 초당 1개 요청만 가능하다.

토큰 버킷 알고리즘은 구현이 비교적 단순하고 실무에서도 많이 사용된다.

### 5.2 누출 버킷 알고리즘

누출 버킷 알고리즘은 요청을 큐에 넣고 일정한 속도로 처리하는 방식이다.

물이 구멍 난 양동이에서 일정한 속도로 빠져나가는 것처럼, 요청도 일정한 속도로 처리된다. 큐가 가득 찬 상태에서 추가 요청이 들어오면 해당 요청은 버려진다.

이 방식은 요청 처리 속도를 일정하게 유지할 수 있다는 장점이 있다. 그러나 순간적으로 많은 요청이 들어왔을 때 큐 대기 시간이 길어질 수 있다.

### 5.3 고정 윈도 카운터 알고리즘

고정 윈도 카운터 알고리즘은 시간을 고정된 구간으로 나누고, 각 구간마다 요청 수를 세는 방식이다.

예를 들어 1분에 100개 요청만 허용한다고 할 때, 12:00:00부터 12:00:59까지 요청 수를 세고 100개를 넘으면 거부한다. 12:01:00이 되면 카운터를 다시 초기화한다.

구현은 단순하지만 경계 문제라는 단점이 있다. 예를 들어 12:00:59에 100개 요청을 보내고, 12:01:00에 다시 100개 요청을 보내면 실제로는 2초 사이에 200개 요청이 들어올 수 있다.

### 5.4 이동 윈도 로그 알고리즘

이동 윈도 로그 알고리즘은 각 요청의 타임스탬프를 저장하고, 현재 시점 기준으로 제한 시간 안에 들어온 요청 수를 계산하는 방식이다.

예를 들어 최근 1분 동안 100개 요청만 허용한다고 하면, 매 요청마다 현재 시점에서 1분 이내의 요청 로그만 남기고 개수를 센다.

이 방식은 정확도가 높다. 그러나 요청마다 로그를 저장해야 하므로 메모리 사용량이 많아질 수 있다.

### 5.5 이동 윈도 카운터 알고리즘

이동 윈도 카운터 알고리즘은 고정 윈도 카운터와 이동 윈도 로그 알고리즘을 절충한 방식이다.

이전 윈도와 현재 윈도의 요청 수를 가중치로 계산하여 현재 요청량을 추정한다. 고정 윈도보다 부드럽게 제한할 수 있고, 이동 윈도 로그보다 메모리 사용량이 적다.

다만 정확도는 이동 윈도 로그 방식보다 낮을 수 있다.

## 6. 책에서의 설계 예시
### 6.1. 처리율 제한 장치 설계 시 요구사항
* 설정된 처리율 초과하는 요청은 정확히 제한해야한다.
* HTTP 응답 시간에 나쁜 영향을 주어서는 곤란하다.
* 가능한 적은 메모리를 사용해야한다.
* 하나의 처리율 제한 장치를 여러 서버나 프로세스에서 공유할 수 있어야한다.
* 예외처리를 하여 사용자에게 제한 사실을 보여주어야한다.
* 제한 장치에 장애가 생기더라도 전체 시스템에 영향을 주어서는 안된다.
> 처리율 제한 장치는 직접 만드는데 시간이 많이 들어가므로 충분한 인력이 없으면 상용 API 게이트웨이 사용하는게 바람직.

### 6.2. 청사진 설계
처리율 제어를 위한 미들웨어를 두고, 요청 카운터는 캐시로 빠르게 동작하는 레디스에 저장하도록 구현하였다.

### 6.3. 상세 설계
#### 1) 처리율 제한 규칙은 어떻게 만들어지고 어디에 저장되는지
설정파일 형태로 디스크에 저장. 작업 프로세스는 수시로 디스크를 읽어 캐시에 규칙을 저장해둔다.
미들웨어는 캐시된 규칙을 읽어 사용.

#### 2) 처리가 제한된 요청들은 어떻게 처리되는지
처리율 제한에 걸린 요청은 일반적으로 HTTP 429 상태 코드로 응답한다.

```text
429 Too Many Requests
```

응답에는 사용자가 언제 다시 요청할 수 있는지 알려주는 헤더를 포함할 수 있다.

대표적인 헤더는 다음과 같다.

```text
X-RateLimit-Limit: 허용 가능한 최대 요청 수
X-RateLimit-Remaining: 남은 요청 수
X-RateLimit-Reset: 제한이 초기화되는 시간
Retry-After: 다시 요청 가능한 시간
```

이런 정보를 제공하면 클라이언트가 불필요한 재시도를 줄일 수 있다.

### 6.4. 분산환경 고려
분산환경 시 발생할 수 있는 문제 두가지.
#### 1) 경쟁조건(Race Condition)
락으로 해결할 수 있지만 성능을 상당히 떨어뜨리는 문제가 있다.
대신 루아 스크립트나 정렬집합이라고 불리는 레디스 자료구조를 사용하여 해결할 수 있다.

#### 2) 동기화
한 대의 처리율 장치 서버로는 충분하지 않아서 여러대 두게 되면 동기화가 필요해진다.
한가지 해결책은 고정 세션을 활용하여 항상 같은 유저의 요청은 같은 처리율 제한 장치로 보낼 수 있도록 하는 것이지만, 
규모 면에서 확장 가능하지 않고 유연하지 않아서 추천하지 않는다고 한다.
더 나은 해결책은 레디스와 같은 중앙 집중형 데이터 저장소를 사용하는 것이다.

## +) 7. 실무에서의 처리율 제한장치 사용 상황
### 7.1. Nginx / API Gateway 레벨
```
Client → Nginx/API Gateway → Application → DB
```
대표적으로 이런 곳에서 처리한다.
```
Nginx limit_req
AWS API Gateway throttling
Kong
Envoy
Spring Cloud Gateway
```
#### 1) Nginx Rate Limit
```
http {
    upstream spring_app {
        server app:8080;
    }
    
    # 클라이언트 IP 기준으로, per_ip라는 이름의 공유 메모리 영역을 10MB 크기로 만들고, 1초당 1개 요청만 허용
    limit_req_zone $binary_remote_addr zone=per_ip:10m rate=10r/s;

    server {
        listen 80;

        location /api/ {
            # burst: 순간적으로 초과 요청을 어느 정도까지 대기열에 넣을 수 있는가
            # nodelay: burst 범위 안의 요청은 지연시키지 않고 즉시 통과.
            limit_req zone=per_ip burst=20 nodelay;

            # 제한 초과시 429 응답
            limit_req_status 429;
            
            proxy_pass http://spring_app;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }
    }
}
```
* 특정 URL 별 rate limit를 걸 수도 있다.


#### 2) AWS WAF의 rate-based rule
AWS WAF의 rate-based rule은 요청을 집계 기준에 따라 세고, 너무 빠른 속도로 들어오면 제한할 수 있는 규칙.
rate-based rule을 ALB나 CloudFront에 붙여서 처리한다.
```
Client
  ↓
AWS WAF
  ↓
AWS ALB
  ↓
Spring App 1
Spring App 2
```

### 7.2. Application 레벨
비즈니스 기준 제한은 애플리케이션에서 처리하는 경우가 많다.
```
회원당 쿠폰 발급 하루 1회
회원당 인증번호 발송 1분에 3회
주문 요청 중복 방지
사용자당 동시 분석 작업 2개
```
이런 건 단순 IP 기준이 아니라 userId, orderId, couponId, fileId 같은 도메인 정보가 필요하기에 
Spring 애플리케이션 내부에서 처리하는게 일반적이다.

#### 1) Redis INCR + TTL
가장 단순하고 흔한 방식.

예를 들어 “사용자당 1분에 10번만 요청 가능”이라면:
```
key = rate-limit:user:1:send-sms
INCR key
EXPIRE key 60초
count > 10이면 차단
```
흐름은 이렇다.
```
1. Redis key 증가
2. 현재 count 확인
3. 제한값 초과 시 429 Too Many Requests 반환
4. TTL이 지나면 다시 요청 가능
```

#### 2) Token Bucket
실무에서 API Gateway나 라이브러리들이 자주 쓰는 방식.
```
토큰이 일정 속도로 채워짐
요청 1개당 토큰 1개 사용
토큰이 없으면 요청 거절
```
장점은 순간적인 burst를 어느 정도 허용할 수 있다는 점이다.

예를들어:
```
평균 초당 10회
순간적으로 최대 30회까지 허용
```

## +) 8. 처리 제한을 적용할 떄 중요한 포인트
### 8.1. 무작정 모든 API에 걸지 않는다.
Rate Limit은 시스템을 보호하지만, 너무 강하게 걸면 정상 사용자 경험이 나빠진다.

그래서 보통 위험도가 높은 API부터 적용한다.

우선순위는 보통 이렇다.

```
1. 로그인
2. 인증번호 발송
3. 비밀번호 찾기
4. 결제/주문
5. 외부 API 호출
6. 검색 API
7. 파일 업로드/분석 요청
```

### 8.2. 제한 초과시 응답을 명확히 준다.
보통 HTTP 429를 사용한다.
```
429 Too Many Requests
```
가능하면 클라이언트가 언제 다시 요청할 수 있는지도 알려준다.
```
Retry-After: 60
```
### 8.3. 모니터링은 중요하다.
운영에서는 이런 지표를 본다.
```
제한에 걸린 요청 수
API별 429 응답 수
IP별 차단 빈도
사용자별 차단 빈도
외부 API 호출량
Redis 부하
```
429가 너무 많으면 제한이 과한 것일 수도 있고, 공격이 들어온 것일 수도 있다.
