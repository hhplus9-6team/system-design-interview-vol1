# 3장: 시스템 설계 면접 공략법

시스템 설계 면접은 알고리즘 면접과 다르게 정답이 존재하지 않는다. 같은 문제를 받더라도 설계자의 경험과 우선순위에 따라 다양한 설계가 가능하다. 따라서 면접관은 특정 기술에 대한 지식보다 문제를 분석하고 설계를 발전시켜 나가는 사고 과정을 더 중요하게 평가한다.

이번 장에서는 시스템 설계 면접을 어떤 방식으로 접근해야 하는지, 그리고 면접관이 실제로 무엇을 확인하려 하는지에 대해 설명한다.

## 시스템 설계 면접에서 평가하는 것

시스템 설계 면접은 단순히 아키텍처를 그리는 시험이 아니다.

면접관은 다음과 같은 역량을 종합적으로 평가한다.

* 요구사항을 구체화하는 능력
* 대규모 트래픽을 고려한 설계 능력
* 기술 선택에 대한 근거 제시 능력
* 확장성과 가용성에 대한 이해
* 커뮤니케이션 능력
* Trade-off를 설명하는 능력

특히 중요한 점은 면접관과 함께 설계를 만들어가는 과정이라는 것이다. 질문 없이 바로 설계를 시작하거나 특정 기술만 나열하는 것은 좋은 평가를 받기 어렵다.

---

## 뉴스 피드 시스템 설계 예시

책에서는 뉴스 피드 시스템을 예시로 시스템 설계 면접 진행 과정을 설명한다.

면접관이 다음과 같은 질문을 했다고 가정해보자.

> 뉴스 피드 시스템을 설계해보세요.

많은 지원자가 바로 데이터베이스나 캐시 설계부터 시작하지만, 시스템 설계 면접의 첫 단계는 설계가 아니라 요구사항 확인이다.

---

## 1단계. 문제 이해 및 요구사항 정의

먼저 문제 범위를 명확하게 해야 한다.

예를 들어 다음과 같은 질문을 할 수 있다.

### 지원자

* 모바일 앱만 지원하면 되나요?
* 일일 활성 사용자 수는 어느 정도인가요?
* 텍스트만 지원하나요, 이미지도 포함되나요?
* 친구 수 제한이 있나요?
* 실시간 피드 갱신이 필요한가요?

### 면접관

* DAU 1,000만 명
* 텍스트 및 이미지 지원
* 평균 친구 수 500명
* 실시간 피드 제공

이러한 과정을 통해 모호한 문제를 구체적인 문제로 변환할 수 있다.

실제 업무에서도 요구사항이 처음부터 완벽하게 정의되는 경우는 드물다. 따라서 좋은 질문을 통해 문제를 명확하게 만드는 능력 자체가 중요한 평가 요소가 된다.

---

## 2단계. 개략적인 설계안 제시

요구사항이 정리되었다면 이제 전체 시스템 구조를 설계한다.

앞서 면접관과의 대화를 통해 다음과 같은 조건을 확인했다고 가정해보자.

* DAU 1,000만 명
* 텍스트 및 이미지 지원
* 평균 친구 수 500명
* 실시간 피드 제공

이러한 조건을 보면 뉴스 피드 시스템은 크게 두 가지 흐름으로 나눌 수 있다.

* 게시물 작성
* 뉴스 피드 조회

### 게시물 작성 흐름

사용자가 게시물을 작성하면 먼저 게시물 데이터와 이미지 정보를 저장해야 한다.

이미지 파일은 데이터베이스에 직접 저장하기보다 객체 저장소(S3 등)에 저장하고, 게시물 정보에는 이미지 URL만 저장하는 것이 일반적이다.

```text
사용자
↓
API Server
↓
Post DB 저장
↓
Object Storage 저장
```

하지만 여기서 끝나지 않는다.

뉴스 피드 서비스는 게시물이 작성되면 친구들의 피드에도 반영되어야 한다.

평균 친구 수가 500명이라고 가정하면 게시물 하나를 작성할 때마다 최대 500개의 피드가 갱신될 수 있다.

이를 API 요청 내부에서 모두 처리하면 응답 속도가 느려질 수 있으므로 일반적으로 비동기 처리를 활용한다.

```text
사용자
↓
API Server
↓
Post DB 저장
↓
Message Queue
↓
Feed Worker
↓
친구들의 피드 갱신
```

메시지 큐를 사용하면 게시물 작성 요청은 빠르게 응답하고, 피드 생성은 별도의 Worker가 처리할 수 있다.

### 뉴스 피드 조회 흐름

사용자가 앱을 실행하면 뉴스 피드를 조회한다.

이때 매번 친구들의 게시물을 모두 조회하고 정렬하는 방식은 비용이 매우 크다.

따라서 미리 생성해둔 피드 정보를 캐시에 저장해두고 활용한다.

```text
사용자
↓
API Server
↓
Feed Cache
↓
Post Cache
↓
Post DB
```

예를 들어 피드 캐시에는 다음과 같이 게시물 ID 목록만 저장할 수 있다.

user:feed:1001

[901, 850, 812, 801, ...]

사용자가 피드를 조회하면 게시물 ID 목록을 가져오고, 실제 게시물 정보는 캐시 또는 DB에서 조회하여 응답한다.

---

## 3단계. 상세 설계

전체 구조가 합의되면 특정 영역을 깊게 분석한다.

뉴스 피드 시스템에서 대표적으로 논의할 수 있는 주제는 피드 생성 방식이다.

### Fan-out on Write

게시물이 작성되는 순간 친구들의 피드 목록에 미리 반영하는 방식이다.

```text
게시물 작성
 ↓
친구 A 피드 저장
친구 B 피드 저장
친구 C 피드 저장
```

장점은 사용자가 피드를 조회할 때 빠른 응답을 제공할 수 있다는 점이다.

반면 친구 수가 많아질수록 게시물 작성 비용이 증가한다.

---

### Fan-out on Read

게시물은 작성자 기준으로만 저장하고, 조회 시점에 필요한 게시물을 모아서 보여주는 방식이다.

```text
피드 조회
 ↓
친구 목록 조회
 ↓
게시물 수집
 ↓
피드 생성
```

장점은 게시물 작성 비용이 낮다는 것이다.

하지만 조회 시 계산 비용이 증가할 수 있다.

---

### 유명인은 어떻게 처리할까?

면접관은 보통 다음과 같은 추가 질문을 한다.

> 친구가 수백만 명인 유명인은 어떻게 처리할 것인가?

일반 사용자에게는 Fan-out on Write를 적용하고,

유명인 계정에는 Fan-out on Read를 적용하는 하이브리드 전략을 사용할 수 있다.

이를 통해 쓰기 성능과 읽기 성능 사이의 균형을 맞출 수 있다.

이처럼 시스템 설계 면접에서는 정답을 말하는 것이 아니라 선택의 이유와 장단점을 설명하는 것이 중요하다.

---

## 4단계. 마무리 및 개선 방향

설계를 마쳤다면 전체 내용을 정리하고 향후 확장 방향을 설명한다.

예를 들어 다음과 같은 내용을 언급할 수 있다.

### 병목 지점

* 데이터베이스 부하
* 캐시 장애
* 대용량 이미지 저장

### 확장 전략

* 데이터베이스 샤딩
* 캐시 서버 확장
* CDN 도입
* 비동기 처리 확대

### 운영 관점

* 모니터링
* 로그 수집
* 장애 대응 전략

실제 서비스는 설계 이후 운영이 시작되므로, 운영 과정에서 발생할 수 있는 문제까지 고려하는 것이 중요하다.

---

면접관은 특정 기술을 알고 있는지보다 문제를 어떻게 분석하고, 어떤 질문을 통해 요구사항을 구체화하며, 여러 선택지의 장단점을 비교하여 설계 의사결정을 내리는지를 보고자 한다.

실제 업무에서도 요구사항을 명확히 이해하지 못한 상태에서 개발을 시작하면 잘못된 방향으로 구현될 가능성이 높다. 그런 점에서 시스템 설계 면접은 단순한 면접 기법이 아니라 실무에서 문제를 해결하는 과정을 압축적으로 보여주는 자리라고 느꼈다.

결국 좋은 시스템 설계자는 빠르게 답을 내리는 사람이 아니라, 좋은 질문을 통해 문제를 정의하고 설계의 근거를 설명할 수 있는 사람이다.
